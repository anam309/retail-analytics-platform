01Extract

Extract & Stage

Pull raw data from source systems into staging. No business logic, no type coercion — this stage only moves rows.

Approach

All source fields land as VARCHAR or TEXT. This prevents silent data loss from premature type casting and surfaces errors explicitly in the Validate stage where they can be tracked.

Metadata to append

  • loaded_at = NOW()
  • source_file — filename or feed identifier
  • batch_date — business date being processed
  • row_num — ordinal for debugging

Load strategy

For small sources (< 500k rows/day): truncate staging then full reload. For large sources: watermark-based append using updated_at >= last_run_ts. Staging tables are transient; they are never the source of truth.

02Validate

Validate & Gate

Two outcomes only: hard reject (row goes to stg_rejections and stops) or pass (row continues). FK misses are not rejections — they are defaults.

— Design principle: Reject when the record itself is invalid (the transaction cannot be trusted). Default to -1 when a lookup simply finds no match (the transaction is real but the master data is incomplete). Never silently drop a row.
Check Category Condition Action Destination
Required transaction fields NULL_VIOLATION transaction_id, product_id, quantity, unit_price, or transaction_date is NULL Reject stg_rejections
Type coercion failure TYPE_ERROR quantity cannot cast to INT; unit_price / discount_amount cannot cast to NUMERIC Reject stg_rejections
Negative or zero quantity RANGE_CHECK quantity <= 0 Reject stg_rejections
Negative unit price RANGE_CHECK unit_price < 0 Reject stg_rejections
Over-discount RANGE_CHECK discount_amount > (quantity × unit_price) Reject stg_rejections
Future transaction date BUSINESS_RULE transaction_date > today Reject stg_rejections
Implausibly old date BUSINESS_RULE transaction_date < '2010-01-01' Reject stg_rejections
Exact duplicate DUPLICATE _stg_sale_id already exists in fact_sales (index scan on uix_fact_sales_stg_id) Skip stg_rejections with resolution_notes='Already loaded'
Unknown customer REF_INTEGRITY customer_id not found in dim_customer WHERE is_current = TRUE Default −1 Continues to fact_sales with customer_sk = -1
Unknown product REF_INTEGRITY product_id not found in dim_product WHERE is_current = TRUE Default −1 Continues to fact_sales with product_sk = -1

Every rejection writes the full source row as JSONB into stg_rejections.raw_record. Operations teams can query, fix, and reprocess without touching the warehouse.

05Facts

fact_sales

Grain: one row per (transaction_id, line_number). FK resolution and all derived metrics happen at load time.

FK resolution — batch lookup pattern

1
customer_sk

Extract all distinct customer_id values from the staging batch. Issue one SELECT customer_id, customer_sk FROM dim_customer WHERE customer_id IN (...) AND is_current = TRUE. Build a Python dict. Map each row; any miss becomes customer_sk = -1. Log miss count to etl_audit_log.

2
product_sk

Same batch-lookup pattern as customer_sk. Build a dict from dim_product WHERE is_current = TRUE. Any miss → product_sk = -1.

3
transaction_date_key

Compute as int(date.strftime('%Y%m%d')). Query dim_date for all distinct date_keys in the batch. Any key missing from dim_date → call fn_populate_dim_date for a ±60 day window around that date. If still missing after population → use sentinel 19000101.

4
Deduplication

Pre-check: filter out any _stg_sale_id already in the unique index uix_fact_sales_stg_id. Then insert remaining rows with INSERT ... ON CONFLICT (transaction_id, line_number) DO NOTHING. After insert, compare rowcount vs input length — the difference is duplicates caught by the grain constraint. Log both counts.

Calculated field mapping

Warehouse field Formula Notes
gross_revenue quantity × unit_price Pre-discount revenue
discount_amount Source field (validated: ≤ gross_revenue) Rejected if over-discount detected in Stage 2
net_revenue gross_revenue − discount_amount Post-discount, pre-tax
cogs quantity × unit_cost unit_cost taken from the current dim_product version at load time — point-in-time accurate for this batch
gross_profit net_revenue − cogs Can be negative for clearance / bundled items
tax_amount Source field NULL-coalesced to 0.00 if absent
total_amount net_revenue + tax_amount What the customer actually paid
06Strategy

Incremental Loading

High-water mark pattern. State lives in etl_audit_log — no external state store required.

fact_sales — watermark extract

Append-only by date

Read the most recent batch_date from etl_audit_log where pipeline_name = 'stg_to_fact_sales' and status = 'SUCCESS'. Extract source records where transaction_date >= last_batch_date. A secondary predicate on source.updated_at >= last_run_ts catches late-arriving corrections.

dim_customer / dim_product — full scan

Compare all, every run

Dimensions are small relative to facts. Running a full hash comparison on every record every day is safer than watermarking — it catches backdated corrections, re-activated records, and out-of-order feeds without special handling. The SCD logic is idempotent for unchanged rows (rows_skipped++).

Idempotency — safe reruns

Every stage can re-run safely

Facts: ON CONFLICT (transaction_id, line_number) DO NOTHING + uix_fact_sales_stg_id. Dimensions: hash comparison skips unchanged rows; UPDATE ... WHERE is_current = TRUE AND hash != :new_hash is a no-op on second run. dim_date: ON CONFLICT (date_key) DO NOTHING.

Failure recovery

Reprocess from last success

On a FAILED or PARTIAL run: read the last status = 'SUCCESS' entry for this pipeline from etl_audit_log. Re-extract from that batch_date forward. Idempotency guarantees already-loaded rows are skipped cleanly — no manual cleanup needed before a rerun.

-- Python: resolve last successful batch date
SELECT batch_date, finished_at
FROM warehouse.etl_audit_log
WHERE pipeline_name = 'stg_to_fact_sales'
  AND status = 'SUCCESS'
ORDER BY started_at DESC
LIMIT 1;

-- Extract from source using that watermark
SELECT * FROM source.orders
WHERE transaction_date >= :last_batch_date
  OR updated_at >= :last_finished_at;
07Audit

Post-load Audit

Runs after fact_sales is loaded. Writes final counts to etl_audit_log and evaluates alert thresholds via vw_dq_daily_load_summary.

Audit log write sequence

1
Open audit record (at pipeline start)

Write a row to etl_audit_log with status = 'RUNNING', started_at = NOW(), pipeline_name, batch_date, and git_commit_sha. Capture the audit_id.

2
Accumulate counts throughout pipeline

Track rows_extracted, rows_inserted, rows_updated (SCD Type 2 expires), rows_rejected, rows_skipped in Python. Increment these per stage.

3
Evaluate DQ thresholds via monitoring view

Query vw_dq_daily_load_summary for today. Raise alerts if: pct_unknown_customers > 5% · pct_negative_margin > 2% · total_rows_loaded < yesterday × 0.5 (volume drop). Also query vw_rejection_summary for open rejection count.

4
Close audit record

Update the etl_audit_log row: set status = 'SUCCESS' (or 'PARTIAL' / 'FAILED'), finished_at = NOW(), final row counts, and error_message if applicable. The duration_seconds column is computed automatically by the generated column definition.

— On unhandled exception: the Python pipeline must catch all exceptions, set status = 'FAILED' and write error_message = str(e) to the audit log before re-raising. A run that crashes without closing its audit record leaves status = 'RUNNING' indefinitely — monitor for stale RUNNING rows older than the expected pipeline duration as a secondary alerting signal.
